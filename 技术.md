# 素材拆解与裂变系统 — 传统技术实现方案

> 不依赖 AI，纯传统技术栈如何实现这套系统？

---

## 一、核心结论

**完全可以不借助 AI 实现这套系统。** 本质上，这套系统的核心是：

1. **结构化数据存储** — 把素材拆成字段存到数据库
2. **模板引擎** — 骨架是模板，裂变是模板填充
3. **规则匹配** — 基于标签和效果数据做推荐排序

这些都是传统 web 开发的标准能力，不需要任何 AI/LLM。

AI 在 P3 阶段才能发挥作用（自动识别素材结构、自动生成变体），但 P1/P2 阶段完全可以用传统技术跑通。

---

## 二、技术选型

### 推荐方案：轻量级全栈

```
┌─────────────────────────────────────────────────────────────┐
│                      前端（纯静态）                           │
│                                                             │
│   HTML + CSS + JavaScript（无框架 或 Vue 3 轻量引入）        │
│                                                             │
│   为什么不用 React/Vue 全家桶？                               │
│   - 系统以表单交互和数据展示为主，不需要复杂状态管理            │
│   - 原型阶段追求开发速度，原生 JS + 少量框架足够               │
│   - 后续可渐进式引入框架                                     │
└──────────────────────────┬──────────────────────────────────┘
                           │ HTTP REST API
┌──────────────────────────▼──────────────────────────────────┐
│                      后端                                    │
│                                                             │
│   Node.js + Express    │    Python + Flask/FastAPI          │
│   （推荐，前后端同语言）  │    （团队熟悉 Python 时选这个）     │
│                                                             │
│   核心依赖：                                                 │
│   - express / fastapi    → Web 框架                         │
│   - mysql2 / pymysql     → 数据库驱动                       │
│   - jsonwebtoken         → 登录鉴权（MVP 阶段可先不做）       │
│   - multer               → 文件上传（批量导入 CSV）           │
└──────────────────────────┬──────────────────────────────────┘
                           │
┌──────────────────────────▼──────────────────────────────────┐
│                      数据库                                  │
│                                                             │
│   MySQL 8.0（推荐）                                         │
│                                                             │
│   为什么选 MySQL？                                           │
│   - 数据关系明确（素材→拆解→骨架→裂变 是标准的关系模型）       │
│   - JSON 字段支持（L3/L4 的结构化数据可以直接存 JSON）         │
│   - 团队熟悉度高，运维成本低                                  │
│                                                             │
│   PostgreSQL 也是好选择，JSON 支持更强，但团队熟悉度优先       │
└─────────────────────────────────────────────────────────────┘
```

### 备选方案对比

| 方案 | 前端 | 后端 | 数据库 | 适合场景 |
|------|------|------|--------|---------|
| **方案A（推荐）** | HTML+JS+Vue3 | Node.js + Express | MySQL 8 | 前后端同语言，开发快 |
| **方案B** | HTML+JS+Vue3 | Python + FastAPI | MySQL 8 | 团队 Python 为主 |
| **方案C（最快）** | 纯 HTML+JS | Python + Flask | SQLite | 单人开发，快速验证 |
| **方案D（企业级）** | Vue3 全家桶 | Spring Boot | MySQL 8 + Redis | 有 Java 团队，长期维护 |

### MVP 阶段最简方案

如果想最快跑通 Demo，推荐 **方案C**：

```
前端：一个 HTML 文件（参考已有 prototype.html）
后端：Python Flask，几十行代码
数据库：SQLite，零配置，一个文件就是整个数据库

开发时间：1-2 天跑通核心流程
```

### 建表 SQL

```sql
-- =============================================
-- 1. 素材表：存储原始素材
-- =============================================
CREATE TABLE material (
    id          INT PRIMARY KEY AUTO_INCREMENT,
    title       VARCHAR(200) NOT NULL COMMENT '素材标题',
    content     TEXT NOT NULL COMMENT '原始素材内容（完整脚本/文案）',
    platform    VARCHAR(50)  COMMENT '平台：抖音/小红书/快手/B站',
    category    VARCHAR(50)  COMMENT '品类：护肤/彩妆/零食/母婴...',
    media_type  VARCHAR(20)  COMMENT '类型：video/image/text',
    source_url  VARCHAR(500) COMMENT '原始链接',
    status      TINYINT DEFAULT 0 COMMENT '0=未拆解 1=已拆解 2=已归档',
    created_by  VARCHAR(50)  COMMENT '录入人',
    created_at  DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at  DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_platform (platform),
    INDEX idx_category (category),
    INDEX idx_status (status)
) COMMENT '素材表';


-- =============================================
-- 2. 拆解表：存储五层拆解结果
-- =============================================
CREATE TABLE dismantle (
    id              INT PRIMARY KEY AUTO_INCREMENT,
    material_id     INT NOT NULL COMMENT '关联素材ID',
    -- L1 主题层
    l1_topic        VARCHAR(200) COMMENT '主题：素材讲什么',
    l1_core_point   VARCHAR(500) COMMENT '核心卖点',
    -- L2 策略层
    l2_strategy     JSON COMMENT '策略标签：["共鸣型","成分党","对比测评"]',
    l2_emotion      VARCHAR(200) COMMENT '情绪策略描述',
    -- L3 结构层（存 JSON，因为段落数量和占比不固定）
    l3_structure    JSON COMMENT '结构段落：[{"name":"开头","function":"痛点共鸣","ratio":15},...]',
    l3_summary      VARCHAR(500) COMMENT '结构一句话描述',
    -- L4 元素层
    l4_elements     JSON COMMENT '元素模块：{"title_formula":"...","hook":"...","transition":"..."}',
    -- L5 表达层
    l5_expressions  JSON COMMENT '具体表达：{"golden_sentences":[],"data_refs":[],"visual_desc":[]}',
    -- 关联骨架
    skeleton_id     INT COMMENT '提取出的骨架ID',
    dismantled_by   VARCHAR(50) COMMENT '拆解人',
    created_at      DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (material_id) REFERENCES material(id),
    INDEX idx_material (material_id),
    INDEX idx_skeleton (skeleton_id)
) COMMENT '拆解表';


-- =============================================
-- 3. 骨架表：可复用的内容框架
-- =============================================
CREATE TABLE skeleton (
    id              INT PRIMARY KEY AUTO_INCREMENT,
    name            VARCHAR(100) NOT NULL COMMENT '骨架名称：测评对比型/红黑榜型...',
    skeleton_type   VARCHAR(50)  COMMENT '骨架类型',
    -- 来源
    source_material_id INT COMMENT '来源素材ID',
    -- 骨架内容
    strategy_desc   VARCHAR(500) COMMENT 'L2策略描述',
    structure_json  JSON NOT NULL COMMENT 'L3结构模板：段落名称、功能、占比',
    elements_json   JSON COMMENT 'L4元素模块模板',
    style_tags      JSON COMMENT '适用风格标签',
    -- 效果统计
    usage_count     INT DEFAULT 0 COMMENT '被使用次数',
    avg_roi         DECIMAL(5,2) COMMENT '平均ROI',
    avg_ctr         DECIMAL(5,2) COMMENT '平均CTR%',
    -- 适用场景
    suitable_for    JSON COMMENT '适用场景：["产品对比","红黑榜","横向评测"]',
    platform        VARCHAR(50) COMMENT '适用平台',
    created_at      DATETIME DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_type (skeleton_type),
    INDEX idx_usage (usage_count DESC),
    INDEX idx_roi (avg_roi DESC)
) COMMENT '骨架表';


-- =============================================
-- 4. 裂变表：记录每次裂变操作和产出
-- =============================================
CREATE TABLE fission (
    id              INT PRIMARY KEY AUTO_INCREMENT,
    skeleton_id     INT NOT NULL COMMENT '使用的骨架ID',
    source_material_id INT COMMENT '母体素材ID（可选）',
    -- 裂变配置
    fission_mode    VARCHAR(50) COMMENT '裂变模式：replace_leaf/replace_branch/replace_style',
    new_topic       VARCHAR(200) COMMENT '新主题',
    new_category    VARCHAR(50)  COMMENT '新品类',
    new_platform    VARCHAR(50)  COMMENT '新平台',
    new_style       VARCHAR(50)  COMMENT '新风格',
    -- 替换内容
    replacement_json JSON COMMENT '替换内容映射：{"L1":"新主题","L5":{...}}',
    -- 产出
    output_title    VARCHAR(200) COMMENT '产出素材标题',
    output_content  TEXT COMMENT '产出素材内容',
    output_status   TINYINT DEFAULT 0 COMMENT '0=草稿 1=待审核 2=已采用 3=已投放',
    -- 预测效果（基于母体数据估算）
    predicted_ctr   VARCHAR(20)  COMMENT '预测CTR范围',
    predicted_roi   VARCHAR(20)  COMMENT '预测ROI范围',
    -- 实际效果（投放后回写）
    actual_ctr      DECIMAL(5,2) COMMENT '实际CTR%',
    actual_roi      DECIMAL(5,2) COMMENT '实际ROI',
    created_by      VARCHAR(50),
    created_at      DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (skeleton_id) REFERENCES skeleton(id),
    INDEX idx_skeleton (skeleton_id),
    INDEX idx_status (output_status)
) COMMENT '裂变表';


-- =============================================
-- 5. 效果数据表：投放数据回写
-- =============================================
CREATE TABLE effect_data (
    id              INT PRIMARY KEY AUTO_INCREMENT,
    material_id     INT COMMENT '关联素材ID',
    fission_id      INT COMMENT '关联裂变素材ID（如果是裂变产出的）',
    platform        VARCHAR(50) COMMENT '投放平台',
    -- 核心指标
    impressions     BIGINT COMMENT '展示量',
    clicks         BIGINT COMMENT '点击量',
    ctr            DECIMAL(5,2) COMMENT '点击率%',
    conversions     INT COMMENT '转化量',
    cvr            DECIMAL(5,2) COMMENT '转化率%',
    cost            DECIMAL(10,2) COMMENT '花费（元）',
    revenue         DECIMAL(10,2) COMMENT '收入（元）',
    roi            DECIMAL(5,2) COMMENT 'ROI',
    cpa            DECIMAL(10,2) COMMENT '单次转化成本',
    -- 时间
    stat_date       DATE COMMENT '数据日期',
    created_at      DATETIME DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_material (material_id),
    INDEX idx_fission (fission_id),
    INDEX idx_date (stat_date)
) COMMENT '效果数据表';


-- =============================================
-- 6. 标签表
-- =============================================
CREATE TABLE tag (
    id      INT PRIMARY KEY AUTO_INCREMENT,
    name    VARCHAR(50) NOT NULL,
    type    VARCHAR(20) COMMENT '标签类型：platform/category/style/strategy',
    UNIQUE KEY uk_name_type (name, type)
) COMMENT '标签表';

CREATE TABLE material_tag (
    material_id INT NOT NULL,
    tag_id      INT NOT NULL,
    PRIMARY KEY (material_id, tag_id)
) COMMENT '素材-标签关联表';
```

## 四、后端 API 设计

### 接口总览

```
素材管理
  POST   /api/material              录入素材
  GET    /api/material              素材列表（分页+筛选）
  GET    /api/material/:id          素材详情
  PUT    /api/material/:id          编辑素材

拆解引擎
  POST   /api/dismantle             提交拆解结果
  GET    /api/dismantle/:materialId 获取拆解结果
  PUT    /api/dismantle/:id         编辑拆解结果

骨架库
  GET    /api/skeleton              骨架列表（按效果排序）
  POST   /api/skeleton              从拆解结果创建骨架
  GET    /api/skeleton/:id          骨架详情
  PUT    /api/skeleton/:id          编辑骨架

裂变引擎
  POST   /api/fission               执行裂变
  GET    /api/fission               裂变历史列表
  GET    /api/fission/:id           裂变结果详情
  PUT    /api/fission/:id           编辑裂变产出

效果数据
  POST   /api/effect                录入/回写投放数据
  GET    /api/effect/:materialId    获取素材效果数据
  GET    /api/effect/compare        对比多个素材效果
```

### 第②步：五层拆解

这一步是**人工填写 + 模板引导**（P1阶段不依赖 AI）。

```
前端：拆解表单页面
─────────────────────────────────────────────────────────
  素材内容预览区（只读，展示原始素材完整内容）
  ─────────────────────────────────────────────────────
  「姐妹们！冬天脸干到爆皮是不是？上妆卡粉卡到怀疑人生？
   我去年踩了8个坑，花了3000多块，才找到这款面霜...」

  【L1 主题层】
  主题：[输入框] 秋冬保湿面霜推荐
  核心卖点：[输入框] 成分好 + 性价比高

  【L2 策略层】
  策略标签：[多选] ☑共鸣型 ☑成分党 ☑对比测评 □悬念型
  情绪策略：[输入框] 踩坑共鸣 → 惊喜发现 → 效果证言

  【L3 结构层】
  ┌──────────┬──────────┬──────┬──────────────────────┐
  │ 段落名称  │ 功能     │ 占比 │ 段落模板              │
  ├──────────┼──────────┼──────┼──────────────────────┤

  │ 主体     │ 成分+感受│ 60%  │ 先看XX——[数据]...     │
  │ 结尾     │ 性价比   │ 25%  │ 下单备注XX送XX       │
  └──────────┴──────────┴──────┴──────────────────────┘

  【L4 元素层】
  标题公式：[输入框] 情绪痛点 + 数字 + 结果
  钩子句式：[输入框] 我踩了N个坑才找到
  转折方式：[输入框] 对比之前用的XX...这款...
  互动设计：[输入框] 评论区告诉我XX，我帮你XX

  【L5 表达层】
  金句：[标签输入] "烟酰胺排第3位" [+]
  数据：[标签输入] "8个坑" "3000多块" [+]
  视觉：[标签输入] 成分表特写 上脸对比 [+]

           [保存拆解结果]  [保存并提取骨架]
─────────────────────────────────────────────────────────
              │
              ▼
后端：POST /api/dismantle
─────────────────────────────────────────────────────────
router.post('/', async (req, res) => {
    const {
        material_id, l1_topic, l1_core_point,
        l2_strategy, l2_emotion,
        l3_structure, l4_elements, l5_expressions
    } = req.body;

    // 1. 保存拆解结果
    const [result] = await db.execute(
        `INSERT INTO dismantle
         (material_id, l1_topic, l1_core_point, l2_strategy, l2_emotion,
          l3_structure, l4_elements, l5_expressions)
         VALUES (?, ?, ?, ?, ?, ?, ?, ?)`,
        [material_id, l1_topic, l1_core_point,
         JSON.stringify(l2_strategy), l2_emotion,
         JSON.stringify(l3_structure),
         JSON.stringify(l4_elements),
         JSON.stringify(l5_expressions)]
    );

    // 2. 更新素材状态为"已拆解"
    await db.execute(
        `UPDATE material SET status = 1 WHERE id = ?`,
        [material_id]
    );

    res.json({
        code: 200,
        data: { dismantle_id: result.insertId }
    });
});
```

### 第③步：提取骨架并入库

这一步是系统的关键——把拆解结果中的**可复用部分**提取为骨架。

```

function generateByTemplate(structure, elements, strategy, newTopic, newExpressions) {
    const exprs = JSON.parse(newExpressions);
    let content = '';

    // 按结构段落逐段生成
    structure.forEach((section, index) => {
        content += `【${section.name} — ${section.function}】\n`;

        switch (section.name) {
            case '开头':
                // 用新主题的痛点替换
                content += `打工人！${newTopic}相关问题是不是？${exprs.hook || ''}\n`;
                break;
            case '主体':
                // 用新数据替换
                if (exprs.data_refs) {
                    content += `先看关键指标——${exprs.data_refs.join('，')}\n`;
                }
                if (exprs.golden_sentences) {
                    content += exprs.golden_sentences[0] + '\n';
                }
                break;
            case '结尾':
                content += `评论区告诉我你的情况，我帮你选！\n`;
                break;
        }
        content += '\n';
    });

    return content;
}


// 效果预测（基于母体骨架的历史数据）
function predictPerformance(skeleton, fission_mode) {
    const baseRoi = skeleton.avg_roi || 2.0;  // 默认基准ROI
    const baseCtr = skeleton.avg_ctr || 1.5;  // 默认基准CTR

    // 不同裂变模式的效果衰减系数
    const modeFactor = {
        'replace_leaf':  0.85,  // 换叶子：效果保留85%（骨架不变，最可靠）
        'replace_branch': 0.65, // 换枝杈：效果保留65%（结构变了，不确定性增加）
        'replace_style':  0.70  // 换表达：效果保留70%（风格变了，受众可能不同）
    };

    const factor = modeFactor[fission_mode] || 0.7;

    return {
        ctr: `${(baseCtr * factor * 0.9).toFixed(1)}%-${(baseCtr * factor * 1.1).toFixed(1)}%`,
        roi: `${(baseRoi * factor * 0.8).toFixed(1)}x-${(baseRoi * factor * 1.1).toFixed(1)}x`
    };
}

## 六、关键算法详解

### 1. 骨架匹配推荐

当用户要裂变时，系统应该推荐**效果最好的骨架**：

```sql
-- 推荐高效果骨架（按品类 + 平台匹配）
SELECT
    s.*,
    -- 综合得分 = ROI权重*0.4 + CTR权重*0.3 + 使用次数权重*0.3
    (s.avg_roi / max_roi * 0.4 + s.avg_ctr / max_ctr * 0.3 +
     s.usage_count / max_usage * 0.3) AS score
FROM skeleton s
WHERE s.platform = '抖音'
  AND JSON_CONTAINS(s.suitable_for, '"产品对比"')
ORDER BY score DESC
LIMIT 10;
```

### 2. 结构相似度判断

判断两个素材是否属于同一骨架类型（用于骨架去重和归类）：

```javascript
// 简单的结构相似度：比较段落名称和功能
function structureSimilarity(structA, structB) {
    const a = JSON.parse(structA);
    const b = JSON.parse(structB);

    if (a.length !== b.length) return 0;

    let matchScore = 0;
    for (let i = 0; i < a.length; i++) {
        // 段落名称匹配得高分
        if (a[i].name === b[i].name) matchScore += 0.6;
        // 功能描述匹配得次高分
        if (a[i].function === b[i].function) matchScore += 0.4;
    }

    return matchScore / a.length;  // 0~1 之间的相似度
}

// 使用：相似度 > 0.7 认为是同一类骨架
// 如果已有同类骨架，可以合并而不是新建
```

## 九、核心矛盾：离散 vs 聚合

这个系统从始至终都在处理一对矛盾——**离散化**和**聚合化**的拉锯。

### 什么是离散？

拆，就是把一个完整的素材打散：

```
完整素材（一篇脚本）
    │ 拆
    ├── L1 主题        ─── 离散出来
    ├── L2 策略        ─── 离散出来
    ├── L3 结构        ─── 离散出来
    ├── L4 元素        ─── 离散出来
    └── L5 表达        ─── 离散出来
```

拆得越细，每一块越**通用**，但也越**零散**。

L3 结构模板可以跨品类复用（面霜→零食），因为它已经被离散成抽象的"痛点→卖点→对比→收尾"框架，不依赖任何具体产品。

### 什么是聚合？

用，就是把离散的零件重新组装成完整素材：

```
L2（策略）+ L3（结构）+ L4（元素）─── 从骨架库聚合
L1（新主题）+ L5（新表达）──────── ─── 新内容填充
                │
                ▼
        完整的新素材（一篇新脚本）
```

聚合的质量取决于：骨架库够不够丰富，标签体系够不够精准。

### 矛盾在哪？

```
        离散度低                          离散度高
        ◄────────────────────────────────────►
        
  素材完整保存                    拆成最小颗粒度
  复制粘贴直接用                  任意组合灵活组装
  复用精度低                      复用精度高
  存储简单                        存储复杂
  人容易理解                      机器容易处理
  
        ▲                            ▲
        │                            │
     传统做法                      理想状态
     （人肉copy）                  （系统化复用）
```

**离散度越高，复用灵活性越大，但管理成本也越高。**

举个极端例子：

- **离散度最低**：直接存整篇脚本 → 只能整篇复制，换个产品就得全改
- **离散度最高**：把每个句子、每个词都拆成独立单元 → 理论上可以组合出无限可能，但组合爆炸，人根本没法审核

五层模型其实是在找一个**平衡点**：

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│   L1 主题    ── 离散度低，一个词/短语就能表达                │
│   L2 策略    ── 离散度低，几个标签就能描述                   │
│   L3 结构    ── 离散度中等，段落框架可跨品类                  │
│   L4 元素    ── 离散度高，标题公式/钩子句式可任意插拔         │
│   L5 表达    ── 离散度最高，每句话都是独立的                  │
│                                                             │
│   裂变时：L1-L4 从骨架库聚合，L5 替换或重填                  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 对技术实现的影响

这个矛盾直接决定了数据库设计和 API 设计：

```
离散 → 数据库需要更多表、更多字段、更多关联关系
聚合 → API 需要更多组装逻辑、更多查询条件

┌──────────────────────────────────────────────────────────┐
│                                                          │
│   拆解存储时 ── 离散                                      │
│   ├── 素材表（存完整内容）                                │
│   ├── 拆解表（存五层结构化数据）                          │
│   ├── 骨架表（存可复用的 L2-L4）                          │
│   ├── 元素表（L4 的进一步拆分）                           │
│   └── 标签表（多维标签）                                  │
│                                                          │
│   裂变查询时 ── 聚合                                      │
│   ├── 按标签筛选骨架（WHERE + JSON_CONTAINS）             │
│   ├── 按效果排序骨架（ORDER BY avg_roi）                  │
│   ├── 骨架 + 新元素组装（模板填充）                       │
│   └── 多骨架批量组合（循环 + 拼接）                       │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

**所以这个系统的技术难点不在于某个单点功能，而在于：**

> **拆的时候怎么存（离散化设计），决定了用的时候怎么取（聚合化查询）。**

数据库 Schema 设计是整个系统的地基——离散粒度定错了，后面聚合查询要么查不出来，要么组合出来的素材质量很差。

这也是为什么 P1 阶段要先人工跑通拆解流程，因为只有实际拆了上百个素材，才能知道离散粒度定得合不合适、哪些字段该合并、哪些该拆分。**Schema 不是设计出来的，是拆出来的。**

## 十、总结：传统技术 vs AI 的边界

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│   传统技术能做的（P1/P2）          AI 才能做的（P3）          │
│   ─────────────────────          ──────────────────          │
│   ✅ 结构化数据存储               ❌ 自动识别素材结构         │
│   ✅ 模板填充裂变                 ❌ 自动生成优质变体         │
│   ✅ 效果数据排序推荐             ❌ 自动匹配最佳骨架         │
│   ✅ 标签检索筛选                 ❌ 语义相似度匹配           │
│   ✅ 人工填写 + 模板引导          ❌ 自然语言理解             │
│                                                             │
│   结论：                                                    │
│   P1/P2 阶段完全不需要 AI                                    │
│   AI 是 P3 的加速器，不是必需品                               │
│   先用传统技术跑通闭环，再用 AI 提升效率                       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**核心就是三件事**：
1. **存** — 把素材按五层结构存入数据库（MySQL + JSON 字段）
2. **拆** — 人工填写拆解模板，提取骨架存入骨架库
3. **裂** — 骨架是模板，新内容是填充物，模板引擎自动组合输出

这就是一个标准的 **CRUD + 模板引擎** 系统，没有任何 AI 也能完整运转。
